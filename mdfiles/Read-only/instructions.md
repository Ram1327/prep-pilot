# INSTRUCTIONS.md

## Purpose

This file defines the mandatory operating procedure for all AI agents working on this project.

Applicable to:

* Hermes
* Gemini Antigravity
* Claude

---

# Startup Procedure

Before performing ANY task:

1. Read PROJECT.md
2. Read CURRENT_STATE.md
3. Read TASKS.md
4. Read SUCCESS_CRITERIA.md
5. Read DECISIONS.md

Only after reading all files may implementation begin.

---

# Required Files

The following files must always exist:

PROJECT.md
CURRENT_STATE.md
TASKS.md
DECISIONS.md
SUCCESS_CRITERIA.md

If any file is missing:

* Create it
* Populate it using project context

---

# PROJECT.md Rules

PROJECT.md contains:

* Project vision
* Tech stack
* Architecture
* Folder structure
* Design philosophy
* Features
* Constraints

Update when architecture changes.

---

# CURRENT_STATE.md Rules

Must contain:

* Current progress
* Current branch
* Current milestone
* Last completed task
* Current blocker
* Next action

Update after every work session.

---

# TASKS.md Rules

Tasks must be broken into:

* Pending
* In Progress
* Completed

Large tasks must be decomposed into smaller tasks.

Never delete completed tasks.

Move them to Completed.

---

# DECISIONS.md Rules

Every major decision must be documented.

Examples:

* Chosen framework
* Database choice
* State management choice
* Authentication strategy
* Deployment strategy

Format:

Date
Decision
Reason
Impact

---

# SUCCESS_CRITERIA.md Rules

Must contain measurable completion criteria.

Example:

Homepage responsive
Authentication working
All forms validated
No console errors
Lighthouse score above 90

Project is NOT complete until criteria are satisfied.

---

# Coding Rules

Always prefer:

1. Readability
2. Maintainability
3. Scalability

Avoid:

* Duplicate code
* Hardcoded values
* Dead code
* Unused files

Follow existing project conventions.

---

# Design Rules

Prefer:

* Premium appearance
* Modern UI
* Consistent spacing
* Responsive layouts
* Accessibility

Every feature should feel production-ready.

---

# Debugging Rules

Before modifying code:

1. Understand root cause
2. Verify assumptions
3. Fix underlying issue

Never blindly patch errors.

---

# Loop Rules

Maximum autonomous loops:

5

Loop structure:

Plan
Implement
Run
Verify
Document

After loop 5:

STOP

Update CURRENT_STATE.md

Wait for user instruction.

---

# Completion Rules

Before ending:

Update CURRENT_STATE.md

Update TASKS.md

Update DECISIONS.md

Verify SUCCESS_CRITERIA.md

Provide concise summary of:

Completed
Remaining
Blockers
Recommended next step

---

# Golden Rule

Do not prioritize speed.

Prioritize correctness, maintainability, and long-term project quality.
